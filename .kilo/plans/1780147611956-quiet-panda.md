# CRUX Polish & Cleanup Plan

## Overview

Post-critical-fixes cleanup across 3 priority tiers. The CRUX theme tokens (green
#22C55E, fonts, brand colors) must remain untouched — the shared
`packages/design-system/tokens.css` has different values (e.g. primary=black) and
will overwrite CRUX if imported naively.

## High Priority (6 items)

### 1. Remove `setMounted(true)` anti-pattern

**Problem:** 8 pages use `setMounted(true)` in a `useEffect` to prevent SSR mismatch,
but `useRequireAuth().ready` already serves this purpose. The dual guard
(`mounted` + `ready`) is redundant.

| File | Current guards | Fix |
|------|---------------|-----|
| `dashboard/page.tsx` | `useRequireAuth().ready` + `mounted` | Remove `mounted`, keep only `ready` |
| `score/[scoreId]/page.tsx` | `useRequireAuth().ready` + `mounted` | Remove `mounted`, keep only `ready` |
| `watch/page.tsx` | `useRequireAuth().ready` + `mounted` | Remove `mounted`, keep only `ready` |
| `cast/page.tsx` | `useRequireAuth().ready` + `mounted` | Remove `mounted`, keep only `ready` |
| `yield/page.tsx` | `useRequireAuth().ready` + `mounted` | Remove `mounted`, keep only `ready` |
| `settings/page.tsx` | `useRequireAuth().ready` + `mounted` | Remove `mounted`, keep only `ready` |
| `lens/[scoreId]/page.tsx` | Inline `useAuth` + `mounted` | Replace inline auth with `useRequireAuth`, remove `mounted` |
| `dashboard/score/page.tsx` | `localStorage.getItem('crux_user')` + `mounted` | Add `useRequireAuth`, remove `mounted` + localStorage guard |

**Approach:** For pages already using `useRequireAuth`, simply delete the
`mounted` state/useEffect and change `if (!mounted || !ready)` → `if (!ready)`.
For `lens/[scoreId]` and `dashboard/score`, first add `useRequireAuth` import
and call, then remove old guards.

**Files:** 8 pages listed above

### 2. StakesAndHorizon.tsx — throttle scroll re-renders

**Problem:** `useParallax()` (lines 32-48) sets `offset` state on EVERY `scroll`
event without throttling. This triggers re-render of the entire component tree
60+ times/second during scroll. On mobile this causes jank.

**Fix:** Wrap the scroll handler in `requestAnimationFrame` or at minimum add
`{ passive: true }` (already present) + throttle via a flag:

```ts
function useParallax() {
  const sectionRef = useRef<HTMLElement>(null);
  const [offset, setOffset] = useState(0);
  const rafRef = useRef<number>(0);

  useEffect(() => {
    const el = sectionRef.current;
    if (!el) return;
    const sectionTop = el.getBoundingClientRect().top + window.scrollY;

    const handleScroll = () => {
      cancelAnimationFrame(rafRef.current);
      rafRef.current = requestAnimationFrame(() => {
        setOffset(window.scrollY - sectionTop);
      });
    };

    window.addEventListener("scroll", handleScroll, { passive: true });
    handleScroll();
    return () => {
      window.removeEventListener("scroll", handleScroll);
      cancelAnimationFrame(rafRef.current);
    };
  }, []);

  return { sectionRef, offset };
}
```

**Files:** `components/sections/StakesAndHorizon.tsx`

### 3. Onboarding: remove `setAnimKey` effect

**Problem:** `onboarding/page.tsx:82-86` has `useEffect(() => setAnimKey(k => k + 1), [step])`
and the content div uses `key={animKey}`. This forces React to unmount/remount
the entire question content on every step change. Since `step` is stable across
steps, we can use `key={step}` directly and delete the effect.

**Fix:**
- Delete lines 82-84 (`const [animKey, setAnimKey] =...` + the effect)
- Change `key={animKey}` → `key={step}` on the content `<div>`

**Files:** `app/onboarding/page.tsx`

### 4. BeforeAfter.tsx — add Fragment keys

**Problem:** Lines 62-92 use `<>...</>` (Fragment shorthand, `<></>`) inside a
`.map()`. JSX shorthand fragments do not accept a `key` prop, causing React to
warn about missing keys for each row.

**Fix:** Replace `<>` with `<React.Fragment key={`row-${i}`}>` and `</>` with
`</React.Fragment>`, or use an explicit `<Fragment>` import.

**Files:** `components/landing/BeforeAfter.tsx`

### 5. Design system tokens import (CRUX-theme-safe approach)

**Problem:** `packages/design-system/tokens.css` contains shadcn-style token
definitions (primary=black, radius=10px, etc.) that conflict with CRUX's green
brand tokens in `globals.css`. The user explicitly noted: **"don't touch the
theme of crux, let it be different from parent one."**

**Analysis:** The shared `tokens.css` defines `@theme inline { ... }` which sets
Tailwind v4 design tokens. If imported in `globals.css`, it will overwrite
CRUX's custom `--color-crux-*` tokens because both use `@theme inline`.

**Fix:** Import `@comfhutt/design-system/tokens.css` **before** the CRUX theme
block. Since `globals.css` processes CSS top-to-bottom and both use `@theme
inline`, the CRUX block (which appears later) will override any conflicting
tokens from the shared package. Non-conflicting tokens (like `:root` CSS
variables for shadcn components) will pass through.

```css
@import "tailwindcss";
@import "tw-animate-css";
@import "shadcn/tailwind.css";
@import "@comfhutt/design-system/tokens.css";  /* ← new: before CRUX theme */
@import "../styles/peeps-animations.css";
@import "../styles/ai-animations.css";

/* ... CRUX brand tokens follow (these win on conflicts) ... */
```

**Decision:** Import before CRUX theme block so CRUX tokens override shared
defaults.

**Files:** `app/globals.css`

### 6. Add `loading.tsx` and `error.tsx` route boundaries

**Problem:** Zero `loading.tsx` or `error.tsx` files exist in the crux app.
During slow API calls (scoring takes ~5s), the user sees a flash of blank page
before skeleton UI appears. If the API dies (backend down), the user gets an
unhandled error.

**Fix:** Add minimal boundaries to key routes:

| Route | loading.tsx | error.tsx |
|-------|------------|-----------|
| `/dashboard` | Skeleton shell with the AppSidebar visible, main area showing shimmer cards | Retry button that calls `router.refresh()`, fallback to dashboard |
| `/lens/[scoreId]` | Chat shell with sidebar + "Loading property data..." spinner | "Score could not be loaded" card with "Back to Dashboard" link |
| `/score/[scoreId]` | Score card skeleton (existing skeleton is already inline — loading.tsx wraps it) | "Report unavailable" card with retry |
| Root `/` (landing) | Not needed (fully static, no data fetching) | Basic error boundary with "Something went wrong" message |

**Implementation:**
- `loading.tsx` files export a loading component matching the page shell layout
- `error.tsx` files use Next.js `'use client'` error boundary with reset button

**Files:** New files: `app/dashboard/loading.tsx`, `app/lens/[scoreId]/loading.tsx`,
`app/lens/[scoreId]/error.tsx`, `app/score/[scoreId]/loading.tsx`,
`app/score/[scoreId]/error.tsx`, `app/error.tsx` (global)

### 9. Replace inline styles with Tailwind in AppSidebar + ProductFamily

**Problem:** `AppSidebar.tsx` (507 lines) and `ProductFamily.tsx` (1702 lines) are
almost entirely inline `style={{}}` objects. This makes them hard to theme,
debug, and maintain. The globals.css already has Tailwind v4 configured with
CRUX brand tokens.

**Scope:**

**AppSidebar** — convert inline styles to Tailwind classes where possible:
- Width transitions (`width: collapsed ? 64 : 260`) → `w-16` / `w-[260px]` with
  transition class
- Colors → `text-crux-text-primary`, `bg-crux-green-tint`, etc.
- Layout → `flex`, `flex-col`, `items-center`, `gap-2`
- Border → `border-r border-crux-border`
- Keep dynamic widths as inline style only where needed

**ProductFamily** — convert non-dynamic styles to Tailwind:
- Static colors, spacing, border-radius → Tailwind utilities
- Keep `motion.div` animations inline (Framer Motion owns those)
- Keep SVG `style` attributes (rendered as presentation attributes)

**Files:** `AppSidebar.tsx`, `ProductFamily.tsx`

**Decision:** Convert both components (full scope).

### 10. Safe-area-inset for mobile tab bar

**Problem:** The mobile tab bar (`dashboard/page.tsx:394-435`) uses `bottom: 12`
with `position: fixed`. On iPhone with home indicator (notch), the tab bar
overlaps the system gesture area. No `env(safe-area-inset-bottom)` anywhere in
the codebase.

**Fix:**
```css
/* In globals.css or inline */
.mobile-tab-bar {
  padding-bottom: calc(12px + env(safe-area-inset-bottom, 0px));
}
```
Or add to the inline style:
```
paddingBottom: \`calc(20px + \${typeof window !== 'undefined' ? 'env(safe-area-inset-bottom, 0px)' : '0px'})\`
```
Simpler: add it as a CSS class in globals.css and apply to the tab bar div.

**Files:** `app/dashboard/page.tsx`, `app/globals.css`

### 16. Peeps SVG filters — verify cross-device rendering

**Problem:** PeepsShield, PeepsGrowth, PeepsSignup use `filter: invert(8%)
sepia(5%) saturate(200%) hue-rotate(180deg) brightness(96%)` via inline style.
CSS filters are universally supported. The Peeps SVGs load from `/peeps/*.svg`
static files — no `<filter>` elements inside SVGs, no `feGaussianBlur` or
`feColorMatrix`. Just standard CSS filters on `<img>` tags.

**Verdict:** No rendering risk. The CSS `filter` property has 97%+ browser
support. No SVG-internal filter elements exist that could break on mobile.

**Fix:** No code changes needed. Add a comment in each Peeps file noting the
filter is intentional for brand consistency.

**Files:** None — documentation only
