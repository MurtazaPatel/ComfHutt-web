# Production Auth Hardening — Frontend + Backend

## Architecture Overview

```
┌────────────────────────────┐     Clerk JWT (Bearer)     ┌───────────────────────────┐
│  Frontend (apps/crux)      │ ──────────────────────────> │  Backend (comfhutt-backend)│
│                            │                             │                           │
│  Clerk nextjs              │                             │  Clerk express             │
│   ├─ SignIn component      │                             │   ├─ clerkMiddleware()     │
│   ├─ SignUp component      │                             │   └─ getAuth(req)         │
│   ├─ useAuth().getToken()  │                             │                           │
│   └─ clerkMiddleware()     │                             │  Legacy: Supabase Auth     │
│                            │                             │   ├─ /api/auth/login       │
│  cruxApi                   │                             │   ├─ /api/auth/register    │
│   └─ Authorization header  │                             │   └─ /api/auth/refresh     │
└────────────────────────────┘                             └───────────────────────────┘
```

## Code Analysis — Issues Found

### Backend (`comfhutt-backend`)

| # | Severity | Issue | Location |
|---|----------|-------|----------|
| A1 | **High** | Dual auth strategy: Clerk JWT (CRUX routes) + Supabase JWT (auth routes). `/api/auth/me` uses Clerk `requireAuth` but would reject Supabase-authenticated users | `src/routes/auth.ts:126`, `src/middleware/requireAuth.ts` |
| A2 | **High** | No CSRF protection on state-changing endpoints | Global — no `csurf`/`csrf-csrf` middleware |
| A3 | **High** | In-memory rate limiting: Cloud Run multi-instance each has its own counter. Malicious actor can cycle IPs/instances to bypass | `src/middleware/rateLimit.middleware.ts` — acknowledged as "post-funding" |
| A4 | **Medium** | No structured auth monitoring: failures logged to `console.error` only. No metrics, no alert thresholds, no Prometheus/GCP Cloud Monitoring | Global |
| A5 | **Medium** | No brute-force protection on auth endpoints: `/api/auth/login` has no account lockout, no progressive delay, no CAPTCHA | `src/routes/auth.ts:52-81` |
| A6 | **Medium** | Token freshness not validated: backend trusts Clerk JWT signature but doesn't check custom claims or session revocation status | `src/middleware/requireAuth.ts` |
| A7 | **Low** | Error handler leaks stack traces in non-production | `src/middleware/errorHandler.ts:21` — intentional, but stack should be sanitized |
| A8 | **Low** | No `PUT`/`PATCH`/`DELETE` methods in CORS — fine for current API but limits extensibility | `src/middleware/cors.ts:20` |

### Frontend (`apps/crux`)

| # | Severity | Issue | Location |
|---|----------|-------|----------|
| B1 | **Medium** | No session persistence across browser tabs: Clerk session is per-tab, no `BroadcastChannel` sync | Global — Clerk handles tab sync internally for v5+ |
| B2 | **Medium** | No graceful error recovery for expired Clerk tokens: `apiFetch` throws generic error with no re-auth trigger | `src/lib/api/index.ts:26-29` |
| B3 | **Medium** | `crux_user` localStorage is separate from Clerk session: user profile cached in localStorage can drift from Clerk's source of truth | `src/app/dashboard/page.tsx:157` |
| B4 | **Low** | No logout cleanup on the backend side: frontend clears localStorage + Clerk signOut but doesn't call `/api/auth/logout` | `src/components/crux/AppSidebar.tsx:416-420` |

### Handshake Issues

| # | Severity | Issue |
|---|----------|-------|
| H1 | **Medium** | Frontend sends Clerk JWT for CRUX endpoints, but backend's `/api/auth/*` routes expect Supabase JWT. Two parallel auth domains in same API. |

## Implementation Plan

### Phase 1 — Backend Security Hardening (Priority: Critical)

#### 1.1 CSRF Protection

**What:** Add `csrf-csrf` middleware that issues a CSRF token on the first GET request and validates it on POST.

```ts
import { doubleCsrf } from 'csrf-csrf';
const { generateToken, doubleCsrfProtection } = doubleCsrf({
  getSecret: () => env.CSRF_SECRET,
  cookieName: 'csrf_token',
  cookieOptions: { sameSite: 'lax', secure: env.APP_ENV === 'production', path: '/' },
});
app.use(doubleCsrfProtection);
```

**Files:** `src/middleware/csrf.ts` (new), `src/app.ts`

#### 1.2 Brute-Force Protection

**What:** Add progressive rate limiting on auth endpoints using `express-rate-limit` with short windows.

| Endpoint | Window | Max | Rationale |
|----------|--------|-----|-----------|
| POST /api/auth/login | 1 min | 5 | Per-IP lockout |
| POST /api/auth/login | 15 min | 20 | Sustained attack ceiling |
| POST /api/auth/login (per-email) | 15 min | 10 | Per-account lockout |

**Files:** `src/middleware/rateLimit.middleware.ts`, `src/services/auth.ts`

#### 1.3 Structured Auth Monitoring

**What:** Add a lightweight monitoring middleware that logs auth events (login success/failure, token validation failure, rate limit hits) to structured logs with request IDs.

```ts
// Track: auth_failure, auth_success, token_invalid, rate_limit_hit
// Log format: { event, userId, ip, timestamp, requestId }
```

**Files:** `src/middleware/authMonitor.ts` (new), `src/app.ts`

#### 1.4 Token Freshness & Revocation Check

**What:** Enhance `requireAuth` to optionally verify the session is still active with Clerk. Currently Clerk's `@clerk/express` handles JWT signature verification automatically. For critical endpoints, add a Clerk API call to verify session status.

```ts
// For critical operations (score compute, watch register):
// Add `verifySession` middleware that calls Clerk's sessions API
```

**Files:** `src/middleware/requireAuth.ts`, `src/middleware/verifySession.ts` (new)

#### 1.5 Unified Auth Strategy

**What:** Phase out the Supabase-based auth routes in favor of Clerk-only auth for all CRUX endpoints. Keep Supabase only for database operations and the `comfhutt-next` website (separate concern).

- Mark `/api/auth/*` routes as deprecated with `@deprecated` comments
- Move all CRUX endpoints to Clerk-only auth
- Remove `requireInternalSecret` dependency from `/api/auth/register`

**Files:** `src/routes/auth.ts`, `src/routes/crux.ts`

---

### Phase 2 — Backend Scalability (Priority: High)

#### 2.1 Redis-Backed Rate Limiting

**What:** Replace in-memory `express-rate-limit` store with Redis (Cloud Memorystore or Upstash). This ensures rate limits are global across all Cloud Run instances.

```ts
import RedisStore from 'rate-limit-redis';
import { createClient } from 'redis';
const redisClient = createClient({ url: env.REDIS_URL });
redisClient.connect();
const rateLimiter = rateLimit({ store: new RedisStore({ sendCommand: (...args) => redisClient.sendCommand(args) }), ... });
```

**Files:** `src/middleware/rateLimit.middleware.ts`, `src/config/env.ts` (+REDIS_URL)

#### 2.2 Response Caching for Public Reads

**What:** Cache GET responses for shareable cards (`/crux/card/share/:token`) and health checks with short TTLs in Redis. Reduces DB load under high traffic.

**Files:** `src/middleware/cache.ts` (new), `src/routes/crux.ts`

#### 2.3 Connection Pooling Optimization

**What:** Verify Supabase/PG connection pool size is appropriate for Cloud Run concurrency. Cloud Run's default is 80 concurrent requests per instance. Pool should be ~20 connections per instance.

**Files:** `src/lib/db.ts`

---

### Phase 3 — Frontend Integration (Priority: High)

#### 3.1 Graceful Token Expiry Handling

**What:** Enhance `apiFetch` to detect 401 responses and trigger Clerk token refresh + retry. Currently fails silently.

```ts
if (res.status === 401) {
  // Signal token refresh to Clerk, retry once
  const newToken = await getToken({ skipCache: true });
  // Retry with new token
}
```

**Files:** `src/lib/api/index.ts`

#### 3.2 Clerk User Sync

**What:** After Clerk sign-in/sign-up, call a backend endpoint to sync Clerk user profile to Supabase `users` table. Ensures `crux_user` localStorage stays consistent with Clerk.

```ts
// POST /api/users/sync — receives Clerk userId + email, upserts in Supabase
```

**Files:** `src/app/signin/[[...rest]]/page.tsx`, `src/app/signup/[[...rest]]/page.tsx` (on success call sync), backend `src/routes/users.ts` (new)

#### 3.3 Logout Cleanup

**What:** On signout, call backend to invalidate any pending watches/sessions in addition to Clerk signOut.

**Files:** `src/components/crux/AppSidebar.tsx`, `src/app/settings/page.tsx`

#### 3.4 Enhanced API Client with Retry

**What:** Add exponential backoff retry for transient failures (503, network error) in `apiFetch`.

**Files:** `src/lib/api/index.ts`

---

### Phase 4 — Security Hardening (Priority: Medium)

#### 4.1 Request Body Validation Hardening

**What:** Add `zod` validation middleware to ALL routes, not just CRUX. Add max length constraints for string fields to prevent DoS via large payloads.

**Files:** `src/middleware/validation.middleware.ts`

#### 4.2 Security Headers Audit

**What:** Review and tighten helmet configuration:
- Add `Cross-Origin-Embedder-Policy`
- Add `Cross-Origin-Opener-Policy`
- Add `Cross-Origin-Resource-Policy`
- Add `Permissions-Policy`

**Files:** `src/middleware/security.ts`

#### 4.3 Session Management Improvements

**What:** 
- Set Clerk session max age (7 days, configurable)
- Add session activity tracking (last active timestamp in Supabase)
- Notify user on new device sign-in

**Files:** Frontend `src/middleware.ts`, Backend `src/services/auth.ts`
